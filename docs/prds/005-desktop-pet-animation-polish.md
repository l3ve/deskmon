# PRD：Deskmon 桌宠动画节奏优化

状态：需求已通过 grill-me 收敛，待实现拆分
目标 triage 标签：ready-for-agent
创建日期：2026-07-08
最后更新：2026-07-08
发布状态：已发布到 GitHub issue #7：https://github.com/l3ve/deskmon/issues/7

## 问题陈述

用户已经有一个能在 macOS 桌面上运行的 Deskmon：宠物可以显示、移动、拖拽、暂停、隐藏、计时、庆祝，也已经换成和应用 icon 更一致的淡紫白小幽灵/软团子角色。

现在的问题不是能力不够，而是动起来的质感还可以继续打磨。桌宠如果只是“贴图平移”，即使功能完整，也会显得像一个漂浮图标，而不是一个有轻微生命感的小像素宠物。用户明确不想在这一轮新增功能，也不想把 Deskmon 扩成工具箱；希望基于现有状态做动画优化，让角色 idle、walk、run、sleep、timer-waiting、celebrate、dragged 这些既有状态更自然。

这次优化必须保持边界克制。它不是新增陪伴系统，不是新增 AI、说话、心情、养成或提醒能力，也不重新生成素材。目标是让现有角色“动得更舒服”，而不是改变 Deskmon 的产品结构。

## 解决方案

对现有桌宠动画和移动节奏做一轮聚焦打磨。保持现有 7 个宠物状态、现有 sprite sheet 布局、现有菜单、现有计时器、现有记忆力能力不变，只调整播放节奏、活动档位参数、移动速度、休息区间、状态切换和必要的庆祝节奏保护。

动画气质以“像素宠物的轻弹跳感”为准。它应该比静态图标更有生命感，但不走流体史莱姆方向，也不做夸张卡通弹簧。像素素材应继续保持硬边，不追求超丝滑动效。

本轮以“标准”活跃度作为主手感校准目标，再让“安静”和“活泼”跟随标准档成比例调整：

- 安静：少移动、慢一点、休息更久，适合不打扰。
- 标准：默认体验，偶尔走走停停，像桌面上自然存在的小伙伴。
- 活泼：移动更频繁、更有冲劲，但不能乱窜或闪烁。

验收以桌面实机观察为主，自动检查为辅。构建通过只能证明没有坏；真正的成功标准是用户在桌面上看它一段时间后觉得角色更自然、更像宠物，而不是被窗口平移拖着走。

## 实现决策

- 本 PRD 是动画打磨，不是新的 Deskmon 能力。
- 实现不能引入新的用户可见功能。
- 实现不能引入新的宠物状态。
- 实现必须保留现有 7 个 mood：`idle`、`walk`、`run`、`sleep`、`timer-waiting`、`celebrate`、`dragged`。
- 实现不能重新生成、替换或重新设计 sprite PNG 素材。
- 实现必须保留现有透明 sprite sheet 契约。
- 实现必须保留当前帧布局：5 个单行动画表和 2 个带方向的双行动画表。
- 实现必须保持像素渲染硬边，不改成平滑插值风格。
- 目标动画气质是像素宠物轻弹跳，不是流体史莱姆模拟，也不是夸张弹簧物理。
- 标准活跃度是主要调参目标。
- 安静和活泼应基于标准手感相对调整，不单独重做。
- 安静模式应优先降低移动频率、降低速度、延长休息窗口。
- 标准模式应成为默认开箱体验下最自然的一档。
- 活泼模式应增加活力，但不能变得不稳定或视觉吵闹。
- walk 应通过移动速度和 sprite 播放节奏匹配，减少滑行感。
- run 应保持明显快于 walk，但避免帧闪得不可读。
- idle 和 sleep 的播放节奏应比移动状态更安静。
- timer-waiting 应表现为耐心等待，不要显得急促或焦虑。
- celebrate 应保持最活泼。
- dragged 应保持有趣且易读。
- 状态切换应避免突兀跳变；在合适处可以用短冷却、休息区间或朝向阈值让切换更自然。
- 移动调参可以调整速度、run 概率、休息区间、到达目标阈值和 mood 选择时机。
- 移动调参不能引入寻路、避障、碰撞、重力、物理反弹或复杂轨迹系统。
- 移动调参必须保留活动区域和可见工作区约束。
- 悬停暂停必须继续停止自主移动，同时保留动画和计时器行为。
- 拖拽必须继续优先于自主移动。
- 计时运行时，timer-waiting 在合适场景下必须继续优先于 walk/run 视觉状态。
- 巨型庆祝行为应保持功能不变，除非需要极小幅度节奏调整来避免视觉突兀。
- 巨型庆祝必须保留现有产品语义：短暂、显眼、不抢焦点、临时。
- 当前设置界面保持不变；不新增动画速度、弹跳或性格控制。
- 当前菜单保持不变。
- 当前记忆力和变量库工作流保持不变。
- 预计不需要新增 native command、权限、持久化 schema 或加密记忆存储改动。

建议拆分的深模块或边界：

- 宠物活动节奏策略：用简洁接口封装安静/标准/活泼的移动速度、run 概率、休息时间和到达目标节奏。
- sprite 播放策略：集中管理 mood 到 FPS 的播放节奏，避免把调参数字散落在控制流里。
- 运动状态协调器：负责 dragged、celebrate、timer-waiting、hover、paused、idle、walk、run、sleep 之间的优先级。
- 巨型庆祝时间线保护：保持已有巨型庆祝时间契约，同时允许小幅视觉顺滑调整。
- 桌面 smoke checklist：记录无法只靠单元测试证明的视觉验收标准。

## 测试决策

- 测试应验证外部行为和模块契约，不测试私有实现细节。
- 本 PRD 的好自动化测试应证明调节奏后，移动和状态优先级仍然正确。
- 自动化测试不应断言精确动画帧号，除非被测对象就是帧布局契约。
- 活动节奏测试应覆盖安静、标准、活泼三个 profile 作为外部可见策略值。
- 活动节奏测试应验证相对行为：安静比标准更克制，活泼比标准更活跃。
- 移动测试应继续覆盖活动区域 clamp 和从活动区域外回归。
- 移动测试应覆盖悬停暂停会停止自主移动，但不禁用动画状态。
- 状态优先级测试应覆盖 dragged 优先于其他 mood。
- 状态优先级测试应覆盖计时运行时 timer-waiting 优先于普通移动 mood。
- 状态优先级测试应覆盖 celebrate 保持临时状态并能回到普通状态。
- sprite 播放检查应确认每个 mood 都映射到有效 sprite sheet 定义。
- sprite 播放检查应确认单行和双行 sprite 布局保持不变。
- sprite 播放检查应确认移动 mood 保持方向行映射。
- 素材 QA 应确认 7 张 sprite PNG 保留 alpha 透明通道和透明四角。
- 构建验证应包含正常前端构建。
- 如果没有改 Rust 侧行为，本 PRD 实现中的 Rust 测试可选。
- 如果触及 native 展示或计时协调，应运行现有 Rust 检查。
- 桌面 smoke test 必须执行，因为视觉运动质量无法完全靠自动化测试覆盖。
- smoke test 应观察 idle、walk、run、sleep、timer-waiting、dragged、celebrate 状态。
- smoke test 应观察安静、标准、活泼三个活跃档位。
- smoke test 应验证右键菜单、拖拽、隐藏、启动计时、取消计时和计时完成仍然可用。
- smoke test 应验证巨型庆祝仍然完整并能恢复。
- smoke test 应验证宠物普通桌面运行几分钟后仍然观感可接受。

## 不在范围内

- 新宠物 mood 或状态。
- 新 sprite 素材。
- 替换和 icon 一致的角色图。
- AI 对话。
- 随机说话气泡。
- 心情、亲密度、喂养或成长系统。
- 新生产力工具。
- 新记忆力或变量库行为。
- 新菜单项。
- 新动画速度、弹跳或性格设置。
- 新计时器选项。
- 计时历史。
- 自定义动画 profile。
- 物理模拟。
- 寻路。
- 避障。
- 屏幕边缘互动。
- 桌面粒子。
- 音效。
- 点击穿透改动。
- native 权限改动。
- 持久化 schema 改动。
- 打包、安装器或应用 icon 改动。

## 进一步说明

- 本 PRD 刻意选择打磨而不是扩张。
- 当前角色素材刚刚与应用 icon 对齐；在考虑任何新素材工作前，应先让这套素材动得更舒服。
- 如果实现过程中发现特定 sprite 缺陷，应单独作为素材 QA 记录，不扩大本 PRD。
- 验收时最核心的产品问题是：在桌面上看它移动一段时间后，Deskmon 是否更像一个小桌面伙伴？
- 实现应足够小，回滚时不影响记忆力、变量库、菜单或 native 存储行为。
