# PRD：Deskmon 轻量专注计时器（连续流程优化）

状态：连续流程已实现，并完成本机主路径验收
目标 triage 标签：ready-for-agent
创建日期：2026-07-09
最后更新：2026-07-10
发布状态：基础能力已由 GitHub issue #9 跟踪并完成；连续流程优化已发布到 GitHub issue #11：https://github.com/l3ve/deskmon/issues/11

## 问题陈述

Deskmon 已经支持三个可配置的专注快捷时长、一个可配置的休息时长、两类完成通知、休息结束声音，以及专注或休息到点后的巨型庆祝。用户可以从菜单栏或宠物右键菜单启动专注，也可以单独启动休息。

现有能力的问题不在于选项不足，而在于一次操作只能启动一段孤立倒计时。专注到点后，计时状态立即清空，宠物庆祝结束后恢复普通状态；如果用户想休息，需要重新打开菜单、重新进入“专注计时”子菜单，再选择休息。休息结束后同样没有承接下一步。功能虽然齐全，但“开始专注、结束专注、进入休息、回到专注”没有形成一套连续的用户流程。

简单地自动开始休息也不能解决问题。用户可能仍在处理当前工作，或者根本没有注意到专注已经结束；如果 Deskmon 在没有确认的情况下开始计算休息时间，倒计时就不能真实表达用户的状态。

本次优化需要把多个孤立计时器收拢成一套轻量的“当前一轮”流程：Deskmon 在需要用户决定时主动把操作放到眼前，但不替用户自动开始休息或下一段专注。整个流程仍然保持轻量，不扩展为任务管理器、完整番茄系统或专注统计工具。

## 解决方案

用户仍然从现有菜单中的三个专注快捷时长开始，不修改宠物左键行为。未开始时不再提供独立休息入口；休息成为专注结束后的明确阶段。

专注进行期间，宠物保持普通大小和原位置，继续使用现有等待动画，不在桌面常驻显示倒计时。用户仍可从菜单查看剩余时间或取消本轮。

专注自然到点后，Deskmon 进入“专注完成待选择”状态：

1. 在鼠标当前所在屏幕的可用工作区中央显示放大的宠物。
2. 即使宠物之前被隐藏，也临时显示该中央交接状态。
3. 宠物播放一轮现有庆祝动画，随后保持放大并切换为安静的普通动画。
4. 宠物旁同时显示一个不自动消失的选择对话框。
5. 对话框持续存在，直到用户明确选择，不自动休息、不自动缩回。
6. 中央交接状态置顶但不抢走当前应用的键盘焦点，也不主动切换桌面空间。

专注完成对话框提供三个操作：

- `休息 X 分钟`：使用本轮开始时锁定的正常休息时长，宠物保持放大居中，对话框原地切换为休息倒计时。
- `再专注 5 分钟`：宠物恢复普通大小、原位置和原可见性，开始固定 5 分钟的额外专注。
- `结束本轮`：结束当前一轮，恢复普通桌宠状态。

选择休息后，宠物持续放大并占据当前屏幕中央，使用现有等待动画。旁边的对话框显示 `休息中 MM:SS` 和 `提前结束休息`。休息从用户点击“休息 X 分钟”时才开始计算，等待选择的时间不计入休息。

休息自然到点后，Deskmon 发送系统通知、按设置播放 macOS 系统声音，并播放一轮庆祝动画；随后宠物继续保持放大居中，对话框切换为三个操作：

- `继续专注 X 分钟`：沿用本轮最初选择的基础专注时长，恢复宠物普通大小、原位置和原可见性后开始专注。
- `再休息 5 分钟`：宠物保持放大居中，开始固定 5 分钟的额外休息。
- `结束本轮`：结束当前一轮，恢复普通桌宠状态。

用户主动选择“提前结束休息”时，不播放声音、不发送通知、不播放庆祝，直接进入同一个休息结束三选一对话框。

额外专注和额外休息都固定为 5 分钟，可以连续重复，不设次数上限。额外专注不会覆盖本轮的基础专注时长，额外休息也不会覆盖本轮的正常休息时长。例如用户从 25 分钟开始：

`专注 25 → 再专注 5 → 休息 X → 再休息 5 → 继续专注 25`

菜单栏菜单和宠物右键菜单必须与中央对话框共享同一流程状态。等待用户选择时，菜单显示和对话框一致的操作，不重新展示普通专注快捷入口；休息中显示剩余时间和“提前结束休息”。用户可以从任一入口完成选择，其他入口立即同步。

这套流程不是自动番茄循环。每一个阶段转换都由用户明确触发，但用户不需要重新寻找功能入口。

## 对话框交互

专注完成：

```text
专注完成
接下来想怎么安排？

[ 休息 X 分钟 ]
[ 再专注 5 分钟 ]
[ 结束本轮 ]
```

休息中：

```text
休息一下
04:59

[ 提前结束休息 ]
```

休息完成：

```text
休息结束
准备好继续了吗？

[ 继续专注 X 分钟 ]
[ 再休息 5 分钟 ]
[ 结束本轮 ]
```

三个选择按钮纵向排列。进入专注完成状态时，“休息 X 分钟”是主操作；进入休息完成状态时，“继续专注 X 分钟”是主操作。对话框不提供关闭按钮，点击对话框外部也不能关闭。

对话框不会主动获取键盘焦点。用户点击对话框后，支持以下键盘操作：

- `Enter` 执行主操作。
- `↑` / `↓` 在按钮间切换焦点。
- `Space` 执行当前按钮。
- `Escape` 不关闭对话框，也不结束本轮。

## 实现决策

- 本 PRD 在已实现的轻量专注设置和计时能力上增加“当前一轮”流程，不回退已有的可配置时长、文案、声音和设置页交互。
- 用户可见命名继续使用“专注计时”。
- 未开始时只提供三个专注快捷入口，移除独立休息入口。
- 不修改宠物左键行为；开始入口仍是菜单栏菜单和宠物右键菜单。
- 当前一轮至少包含：基础专注时长、正常休息时长、通知文案、声音策略、原可见性、原位置和当前流程状态。
- 开始一轮时锁定当前设置；设置变化只影响下一轮。
- 流程状态集中建模为：未开始、专注中、专注完成待选择、休息中、休息完成待选择。
- 计时段和流程状态分开建模：基础专注、额外专注、正常休息、额外休息都是计时段，但共同属于当前一轮。
- 所有状态转换通过一个集中流程模块完成，菜单事件、计时自然完成和宠物对话框不能各自直接改写部分状态。
- 流程模块需要返回稳定快照，供菜单和宠物前端共同渲染。
- 专注中取消等同于结束本轮，不发送通知、不播放声音、不庆祝。
- 正常休息从用户确认时开始，不把专注完成后的等待时间计入休息。
- 额外专注和额外休息固定为 5 分钟，不新增设置项。
- 额外计时可以连续重复，不设次数上限，不记录次数。
- 专注完成和休息完成都保留系统通知；只有休息自然完成按当前设置播放系统声音。
- 提前结束休息直接进入休息完成待选择状态，不触发自然完成反馈。
- 专注完成时播放一轮庆祝，随后保持放大并切换到安静动画。
- 休息中使用现有等待动画；休息自然完成时再播放一轮庆祝，随后保持放大等待选择。
- 不新增宠物 sprite 素材或新的宠物动画状态。
- 中央交接复用现有巨型庆祝的自适应尺寸规则，不新增尺寸设置。
- 中央对话框是宠物临时呈现的一部分，不创建常驻专注窗口，不进入 Dock 窗口管理模型。
- 中央呈现需要为宠物和对话框共同提供稳定布局；对话框优先在宠物右侧，空间不足时切换到左侧或下方。
- 中央呈现位于鼠标当前所在屏幕的可用工作区中央；出现后不跟随鼠标跨屏移动。
- 如果显示器断开，中央呈现重新约束到当前可用工作区。
- 中央呈现置顶但不主动聚焦，也不主动切换桌面空间。
- 完成状态无超时，不自动缩回，不允许点击外部关闭。
- 中央呈现期间禁用宠物拖拽；右键菜单保持可用。
- 如果计时到点时正在拖拽宠物，先记录完成状态，等拖拽结束后立即进入中央交接，不能跳过本轮交接。
- 宠物隐藏时到点也显示中央交接；重新进入专注或结束本轮后恢复进入交接前的可见性。
- 重新进入专注或结束本轮时恢复普通尺寸、原位置和用户的默认置顶偏好；位置失效时约束到可见工作区。
- 休息场景中原“隐藏宠物”入口改为“结束本轮并隐藏宠物”。
- 菜单栏菜单、宠物右键菜单和中央对话框共享同一流程快照和动作语义。
- 计时中的剩余时间继续通过更新已有菜单项文本刷新，避免每秒重建已打开的原生菜单。
- 等待选择时，菜单只显示与中央对话框一致的流程操作，不显示普通专注快捷入口。
- 对话框按钮使用原生桌面式焦点顺序和可访问名称，不使用只有图标而没有语义的操作。
- 对话框不提供关闭按钮；结束必须使用“结束本轮”。
- 对话框未获焦点前不响应快捷操作；获焦点后支持方向键、Enter 和 Space，Escape 不关闭。
- 当前一轮只存在于内存；主动退出或应用重启后不恢复。
- 当前一轮不写入历史，不生成完成计数，不修改设置数据结构中的长期统计。

建议拆分的深模块或边界：

- 专注流程状态机：接收开始、自然到点、取消、选择休息、追加专注、提前结束休息、继续专注、追加休息和结束本轮事件，返回单一权威快照。
- 计时段执行器：只负责当前计时段的结束时间、剩余时间和自然完成事件，不自行决定下一阶段。
- 菜单映射器：根据流程快照生成菜单项和动作映射，并复用现有剩余时间原地更新能力。
- 临时呈现协调器：管理当前屏幕选择、放大居中、原状态保存与恢复、隐藏例外、显示器变化和拖拽结束后的延迟交接。
- 宠物交接对话框：根据流程快照渲染专注完成、休息中和休息完成三种界面，并只发出语义化动作。
- 完成反馈策略：区分专注自然完成、休息自然完成和休息主动提前结束，集中决定通知、声音和庆祝。

## 测试决策

- 测试应验证用户可见行为和模块契约，不依赖线程实现、DOM 层级或私有字段。
- 专注流程状态机需要作为主要单元测试边界，覆盖所有合法状态转换和非法动作不改变状态。
- 状态机测试应覆盖从三个专注快捷入口开始一轮。
- 状态机测试应覆盖未开始时不能直接进入休息。
- 状态机测试应覆盖专注自然完成进入专注完成待选择，而不是回到未开始。
- 状态机测试应覆盖专注中取消直接结束本轮。
- 状态机测试应覆盖正常休息从用户确认时开始。
- 状态机测试应覆盖额外专注固定 5 分钟且不覆盖基础专注时长。
- 状态机测试应覆盖额外休息固定 5 分钟且不覆盖正常休息时长。
- 状态机测试应覆盖额外专注和额外休息可以连续重复。
- 状态机测试应覆盖休息自然完成和提前结束都进入休息完成待选择，但完成原因不同。
- 状态机测试应覆盖继续专注沿用基础专注时长。
- 状态机测试应覆盖结束本轮清空当前流程上下文。
- 状态机测试应覆盖开始一轮后修改设置不影响本轮锁定值。
- 计时段执行器测试应覆盖剩余时间归零只产生一次自然完成事件。
- 完成反馈策略测试应覆盖专注完成有通知、无声音。
- 完成反馈策略测试应覆盖休息自然完成有通知，并根据开关决定是否播放声音。
- 完成反馈策略测试应覆盖提前结束休息无通知、无声音、无庆祝。
- 菜单映射测试应覆盖五种流程状态对应的菜单项、文案和动作。
- 菜单映射测试应覆盖未开始时只有三个专注入口，没有独立休息入口。
- 菜单映射测试应覆盖两个待选择状态与中央对话框操作完全一致。
- 菜单映射测试应覆盖休息中使用“提前结束休息”，专注中使用“取消本轮”。
- 菜单刷新测试应继续确认运行中只更新已有剩余时间项，不每秒重建整棵菜单。
- 临时呈现测试应覆盖使用鼠标所在屏幕、复用巨型尺寸、出现后不跟随鼠标和显示器断开后的重新约束。
- 临时呈现测试应覆盖保存并恢复原大小、位置、置顶偏好和可见性。
- 临时呈现测试应覆盖宠物隐藏时到点仍显示，重新进入专注后恢复隐藏。
- 临时呈现测试应覆盖拖拽中到点延迟到拖拽结束后交接。
- 对话框交互测试应覆盖无关闭按钮、点击外部不关闭、按钮焦点顺序和 Escape 不结束本轮。
- 桌面 smoke test 应覆盖完整主路径：开始专注、自然到点、开始休息、休息自然到点、继续专注、结束本轮。
- 桌面 smoke test 应覆盖额外专注、额外休息和提前结束休息。
- 桌面 smoke test 应覆盖菜单与中央对话框双向同步。
- 桌面 smoke test 应覆盖隐藏宠物、多显示器、中央状态右键菜单和“结束本轮并隐藏宠物”。
- 桌面 smoke test 应确认中央呈现不会抢走其他应用的键盘焦点。
- 现有设置校验、通知文案、系统声音和紧凑步进器测试继续保留，避免基础能力回归。

## 不在范围内

- 自动开始休息。
- 自动开始下一轮专注。
- 无需用户确认的循环番茄钟。
- 独立休息快捷入口。
- 长休息和四轮番茄规则。
- 专注任务、任务列表或任务绑定。
- 今日番茄数、完成轮数、连续次数和累计时长。
- 专注历史、休息历史和统计导出。
- 跨设备同步。
- 当前一轮跨应用重启恢复。
- 延后提醒、重复提醒和稍后再问。
- 新的常驻专注窗口。
- 修改宠物左键行为。
- 专注期间常驻倒计时气泡。
- 与当前流程无关的宠物对话或文字气泡。
- 新宠物状态、新角色形象或新 sprite 素材。
- 宠物动作音效、自定义铃声和音频导入。
- 全局快捷键。
- 菜单栏图标旁常驻倒计时文本。
- 记忆力、笔记本或变量库改动。

## 进一步说明

- 旧版“专注结束后直接回到未计时状态”的决定被本次 PRD 明确替换；它是现有操作割裂的主要原因。
- 本次新增的是需要用户明确处理的流程交接，不是自动推进。Deskmon 把下一步放到用户眼前，但不会替用户判断是否已经开始休息。
- 中央对话框属于桌宠的临时交互状态，不是新的管理窗口。它只在专注完成、休息中和休息完成时出现。
- 休息时保持放大居中是有意的产品表达：用户选择休息后，Deskmon 用一个持续可见的桌面场景承载这段恢复时间。
- 原 GitHub issue #9 和对应实现继续作为基础能力记录；本次连续流程优化应由新的 issue 跟踪，避免重新打开已经完成的交付。
